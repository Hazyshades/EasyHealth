# Design Critique: EasyHealth In-App Pages

**Scope:** All authenticated app routes under `src/app/app`, onboarding flows, the check-email screen, and legal pages. The marketing landing page (`src/app/page.tsx`) was excluded per request.

## AI Slop / Anti-Pattern Verdict

**LLM assessment:** The app UI does not read as obviously AI-generated. It follows a consistent shadcn/Tailwind vocabulary and a single type family. The biggest tells are small local breaks from the system: onboarding screens use a bright `#2563eb` (brand-indigo mismatch), `SurfaceCard` pairs a 1px border with a 24px-blur shadow, `rounded-[18px]` cards exceed the product-card ceiling, and several pages fall back to plain text loaders instead of skeleton states.

**Deterministic scan:** The bundled detector found **7 advisory** issues, all `design-system-font-size` literals: `10px` in dashboard widgets and the filter menu, and `11px` in `biomarker-table.tsx` for specimen/meta text. These are functional micro-labels; they are smaller than the documented label ramp (`0.75rem`/12px) and should be moved to a named micro-label token if they stay, or adjusted to the label size.

**Browser overlay:** Not available. Authenticated routes redirect to `/`, so the live detector could not be injected. Findings rest on static source analysis.

## Overall Impression

The product side of EasyHealth is already usable and calm. It is not alienating. The strongest work is in trustworthy error handling and clear document status. The weakest area is consistency: several pages were built at different moments and no longer share the same color, radius, or loading vocabulary. The interface also under-serves mobile users and first-time health data consumers who need more context for unfamiliar terminology.

## What's Working

1. **Trustworthy status language.** "Processing," "Queued," "Ready," and the type-mismatch banner keep users oriented without implying diagnosis.
2. **Clear navigation scaffold.** Sidebar + top-bar + mobile bottom nav create a consistent wayfinding frame with visible active states and focus rings.
3. **Good error recovery.** Error states on account, report detail, document viewer, and onboarding provide plain text, a retry/back path, and preserve user work.

## Priority Issues

### [P1] SurfaceCard couples 1px border with a wide diffuse shadow
- **What:** `SurfaceCard` uses `border border-[var(--eh-border)]` plus a shadow with blur up to 24px (`0_8px_24px_-8px...`). This is the banned "ghost-card" pattern.
- **Why it matters:** It makes the app default look fuzzy and off-brand; it also conflicts with the "border OR small shadow" rule in the design system.
- **Fix:** Switch `SurfaceCard` to either a clean 1px border + flat background, or a single small structural shadow ≤ 8px. Remove the hover lift if it stays border-only.
- **Command:** `$impeccable polish src/components/ui/surface-card.tsx`

### [P1] Onboarding flows use a different primary color than the app
- **What:** The profile and consent onboarding forms use `bg-[#2563eb]` (bright Chrome blue), while the authenticated app uses `#4f46e5` indigo.
- **Why it matters:** Users pass from landing → onboarding → app. A sudden hue shift subtly undermines the "trustworthy/calm" personality and breaks the One Accent Rule.
- **Fix:** Replace `#2563eb` with `bg-[var(--eh-brand)]` on `src/app/onboarding/profile/page.tsx` and `src/app/onboarding/consent/page.tsx`.
- **Command:** `$impeccable colorize onboarding`

### [P2] Plain-text loaders instead of skeletons
- **What:** Many pages (`account`, `documents`, `dashboard`, `AI settings`, `onboarding/profile`, `onboarding/consent`) return `Loading…` text while data is fetched.
- **Why it matters:** Skeletons preserve layout and reduce perceived wait time; text-only spinners make the app feel unfinished.
- **Fix:** Introduce skeleton variants for `SurfaceCard`, `DataTable`, and empty page headers, and use them as fallbacks in `@/app/app/**/page.tsx` and onboarding pages.
- **Command:** `$impeccable harden loading`

### [P2] Reports/create uses a hand-rolled, non-portalled modal
- **What:** The document-selection modal in `src/app/app/reports/create/page.tsx` is a `fixed inset-0` overlay implemented inline, not via the project’s dialog/dropdown primitives.
- **Why it matters:** It risks z-index clipping in the app shell, lacks standard focus trapping/scroll lock, and duplicates layout decisions.
- **Fix:** Refactor it to the existing `Dialog`/`Sheet` primitive or repurpose the `select`able list inline.
- **Command:** `$impeccable harden reports/create`

### [P2] Status chips rely on color + text but no shape/icon
- **What:** `StatusChip` shows only text inside a colored pill.
- **Why it matters:** The Product Principle is "Do not rely on color alone." The current chips pass a minimal bar because text is present, but color-blind users and low-lit screens can still struggle.
- **Fix:** Add a 12×12 dot or Lucide icon to each variant (`Check`, `AlertCircle`, `Loader2`, `Info`) so status is recognizable without reading hue.
- **Command:** `$impeccable harden status-chip`

### [P3] Typography ramp has undocumented 10px/11px literals
- **What:** Dashboard widgets, biomarker-table metadata, and the filter menu use literal `text-[10px]` and `text-[11px]`.
- **Why it matters:** They are outside the system; future components will copy them by example.
- **Fix:** Add a `micro` or `caption` step to the design system and replace all literals with that token.
- **Command:** `$impeccable typeset labels`

### [P3] Some settings/account pages lack a visible page title
- **What:** `AI settings` and `Account` render `PageHeader` with only `subtitle` (no `title`).
- **Why it matters:** Screen-reader users and navigators lose the page landmark; the TopBar title resolves from path, but the content area has no `h1`.
- **Fix:** Always pass `title` to `PageHeader`.
- **Command:** `$impeccable layout settings`

## Persona Red Flags

### Alex (Power User)
- **No shortcuts:** Cannot bulk-select documents, bulk-delete, or export data.
- **No saved filters:** Documents and biomarkers reset filters on every visit.
- **No keyboard navigation for the body/health map:** Interactive zones are likely mouse-first.
- **Verdict:** Will tolerate the app but leave if daily review becomes click-heavy.

### Jordan (Health-Anxious First-Timer)
- **Terms without help:** "Out-of-range only," "Confidence," "Normal vs Attention," "Abnormal" appear with no inline explanation.
- **Clinical-looking tables:** The biomarker table shows ranges but not what they mean or what to do.
- **Processing wait without reassurance:** "Loading document…" does not indicate estimated time or that the original file is safely stored.
- **Verdict:** Anxiety spikes at exactly the moments the product promises to calm.

### Sam (Accessibility-Dependent User)
- **Icon-only Sign-out button on small screens:** The TopBar shows the LogOut icon + `sr-only` label? It actually shows the icon always and hides the text on small screens, but the `<Button>` itself lacks an `aria-label` (the visible text is hidden on mobile).
- **Status chips lack icons:** Non-sighted users get the label text, but sighted low-vision users miss the quick shape cue.
- **Onboarding checkboxes use browser defaults:** They work, but focus styling is only `accent` color, not a visible ring.
- **Verdict:** Likely usable with screen readers, but mobile and status scanning feel fragile.

## Minor Observations

- The upload page uses `text-teal-700` for the back link; it is health-context appropriate but sits outside the explicit brand link color.
- Check-email page is isolated from the app shell and has a warm teal gradient background that differs from the app shell; acceptable as a standalone auth surface, but it should still use `eh-brand` for primary action if it re-enters later.
- Data-table headers use uppercase/tracking, which is acceptable for column labels but should not spread to section eyebrows.
- Active nav item uses a strong indigo shadow; on light mode it is the only saturated surface and reads as the brand beacon — good when used once per page.

## Questions to Consider

1. Should onboarding forms share the exact same primary button component as the app, or do they need a separate "standalone" style?  
2. What if the document viewer had a persistent "How was this extracted?" help panel to reduce anxiety for first-time users?  
3. Could the empty states (no documents/biomarkers) include an educational preview of what the user will see after uploading?
